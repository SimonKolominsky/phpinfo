REST API (Representational State Transfer Application Programming Interface) — это архитектурный стиль для создания веб-сервисов, основанный на стандартах HTTP и принципах, обеспечивающих простоту, масштабируемость и надежность взаимодействия между клиентом и сервером. Вот ключевые аспекты:

Основные принципы REST:

  1. Ресурсы и URI (Uniform Resource Identifier)
    Каждый ресурс (например, пользователь, статья) идентифицируется уникальным URI (например, /users, /posts/123).
    → Используются существительные в URL, а действия определяются HTTP-методами.

  2. HTTP-методы (CRUD операции)

    • GET: Получение ресурса (например, GET /users/1).

    • POST: Создание ресурса (например, POST /users).

    • PUT/PATCH: Обновление (полное/частичное, например, PUT /users/1).

    • DELETE: Удаление (например, DELETE /users/1).

  3. Статус-коды HTTP
    Ответы включают стандартные коды:

    • 200 OK (успешный GET),

    • 201 Created (успешный POST),

    • 404 Not Found (ресурс не найден),

    • 500 Internal Server Error (ошибка сервера).

  4. Статус-независимость (Stateless)
    Каждый запрос содержит всю информацию для его обработки. Сервер не хранит состояние клиента.

  5. Кэширование
    Ответы помечаются как кэшируемые или нет, чтобы уменьшить нагрузку на сервер.

  6. Единый интерфейс

    • HATEOAS (Hypermedia as the Engine of Application State): Ответы могут содержать ссылки на связанные ресурсы (например, "links": {"self": "/users/1", "posts": "/users/1/posts"}).

  7. Слоистая архитектура
    Клиент не знает, взаимодействует ли он напрямую с сервером или через промежуточные узлы (прокси, балансировщики нагрузки).

Примеры использования:

  • Мобильное приложение запрашивает данные о погоде через GET /weather?city=London.

  • Веб-форма отправляет данные пользователя через POST /users.

  • Обновление профиля через PUT /users/42.

Форматы данных:

  • JSON (наиболее популярный), XML, HTML, текст.

Практические советы:

  • Версионирование API: Например, /api/v1/users.

  • Аутентификация: Через токены (OAuth 2.0, JWT) или API-ключи в заголовках.

  • Документация: Используйте стандарты OpenAPI (Swagger).

REST vs RESTful:

  • RESTful — строгое соблюдение всех принципов REST (включая HATEOAS).

  • REST — часто используется для API, которые частично следуют принципам (например, без HATEOAS).

Преимущества:

  • Масштабируемость за счет статичности и кэширования.

  • Гибкость благодаря поддержке JSON/XML и HTTP-стандартов.

  • Универсальность — подходит для веб, мобильных приложений, IoT.

REST API стал стандартом для интеграции веб-сервисов благодаря простоте и совместимости с существующей инфраструктурой.
